# Part 1: General Guidelines

## §1 Voice and Tone

### 1.1 Brand Voice — Calm · Anatomical · Accountable

Humanbased copy reads like a well-kept lab notebook: warm, precise, nothing decorative.

**Calm** — No hype, no urgency theater. State what's true. Let the substance carry the weight. The product has real stakes (real people, real labor, real attribution) — that doesn't need amplifying.

**Anatomical** — Name things exactly. Use the right word, not the impressive-sounding one. Prefer nouns and action verbs over adjective stacks. If you can't describe it specifically, you don't understand it yet.

**Accountable** — Say what happened, what it means, and what comes next. No vague gestures toward outcomes. No "we're excited to announce" without a concrete follow-through. Own the claim.

### 1.2 Tone (adjusts per content piece)

- Brief and punchy — clarity over completeness in short-form
- No technical jargon or corporate talk
- Direct — "offers" not "aims to offer"
- Addresses challenges directly; stays solutions-oriented
- Not casual, but conversational

### 1.3 What to avoid

| ❌ | Why |
|---|---|
| "Join the revolution." | Empty — no substance behind it |
| "The most advanced platform for…" | Unverifiable superlative |
| Emoji storms 🚀🚀🚀 | Undermines precision |
| Machine IDs surfaced to readers | Adds noise, no meaning |
| Passive voice by default | Buries agency and responsibility |

## §2 Clarity and Flow

> **Scope: social media posts only.**

### 2.1 Clarity

- Use transition words and position clauses deliberately; favor action verbs over be-verbs (*is / are*).
- Plain style: use `use` not `utilize`. Default to the simpler word.
- Active over passive in most cases.
- Two short sentences beat one long one.

### 2.2 Flow

- Avoid sentence fragments.
- Vary sentence structure, rhythm, opening words, and length.

## §3 Nuances

### 3.1 Canonical Terminology

**Use these words:**

| Use | Not |
|---|---|
| contributor | worker, user, talent |
| campaign | project, task group |
| frontier | (keep in English in technical contexts) |
| provenance / lineage | (keep in English for attribution contexts) |
| accepted | approved, passed (for task status) |
| instance | (minimum contribution unit, technical docs only) |
| onchain | on-chain, on chain |

**Brand name formatting:**

| Item | Format |
|---|---|
| Product name | **Humanbased** — one word, no hyphen, no space, no mixed casing |
| URL / domain | **humanbased.ai** — always lowercase |
| Token | **XnY** / **$XNY** |
| Legal entity | **Outtrain AI Limited** — in legal documents and payment copy |

### 3.2 Capitalization

- Follow [Purdue OWL capitalization guidelines](https://owl.purdue.edu/owl/general_writing/mechanics/help_with_capitals.html) as the baseline
- Titles use **Title Case** (not sentence case)
  - e.g. *"Onchain Summer II is Coming"*

### 3.3 Punctuation

- **Oxford Comma** — always use it
  - ❌ apples, oranges and bananas
  - ✅ apples, oranges**,** and bananas
- **Em Dashes** — written as `—` with **one space on each side**
  - e.g. *"Facilitators accept responsibility to move groups through an agenda — if necessary — suggest alternates."*
- **Ampersand (&)** — only in proper names (`Tiffany&Co.`) and common abbreviations (`R&D`). Do not substitute for "and"
- **Parentheticals** — when a sentence ends with a parenthetical, the period goes **outside** the closing parenthesis (unless the entire sentence is inside)
- **Colons & Semicolons** — use semicolons to separate list items when at least one item contains a comma; otherwise use commas

### 3.4 Numbers, Dates, and Times

- **Numbers:**
  - Spell out 0–9 (zero, one, two … nine)
  - Exception: `1-for-1`, not `one-for-one`
  - 10 and above: use numerals (10, 59, 100…)
  - 10,000 and above: use `K` — write `100K`, not `100,000`
  - 1,000,000 and above: use `M`
  - 1,000,000,000 and above: use `B`

- **Dates:** write as "September 2, 2022" or "Sep. 2, 2022" — not "September 2nd, 2022"

- **Year ranges:** "2009—2022" (em dash, no spaces)

- **Decades:** "1960s" on first use, then "60s" — no apostrophe

- **Times:**
  - Always include period + timezone; period lowercase, timezone uppercase
  - No space between numeral and period; one space before timezone
  - e.g. "7am ET", "1:30am PT"
  - Default to PT when location is unspecified

- **Software versions:** lowercase prefix — `v1`, `v1.5`, `v2.6.4` (not `V1`)

# Part 2: Base-Specific Rules

## §4 Additional rules when writing Base-related content

> **Scope: applies only when writing content that involves Base** — joint campaigns, Base ecosystem posts, @base replies/reposts, Humanbased × Base co-branded content. **Stack on top of §1–§3.**
>
> Following these rules directly affects whether Base's core team will amplify the post — one of the most important PR levers in the current Base ecosystem push (`../10_PR/Base/`).

### 4.1 Canonical description of Base

> Base is a secure, low-cost, builder-friendly Ethereum L2 designed to help bring a billion users and a million developers onchain with an ecosystem of onchain applications.

Align to this phrasing when introducing what Base is.

### 4.2 Base naming and formatting rules

| Item | ✅ Use this | ❌ Not this |
|---|---|---|
| Base's name | **Base** | BASE / Base Chain / $BASE / Base Network |
| Onchain context | **onchain** / Onchain (one word) | web3 / crypto / on-chain / on chain / On Chain |
| Brand order | **Humanbased × Base** | Base × Humanbased |
| Relationship language | "launched on / built on / integrated with Base" | "partnership" / "collaboration with Base" (unless formally approved) |
| Post focus | Connect to Base's mission | Feature-only or milestone-only posts |

### 4.3 Six writing principles for Base content

| Principle | How to apply |
|---|---|
| **Emphasize Base's vision and mission** | Highlight "bring a billion people onchain", "global onchain economy", "innovation, creativity, freedom". Bring Humanbased content back to this throughline. |
| **Focus on progress and next steps** | Celebrate recent milestones; state upcoming priorities; invite community participation. |
| **Use specific examples, not abstract concepts** | Avoid "web3" / "blockchains" as buzzwords. Land on concrete specifics: affordable transactions, seamless onboarding, simple developer tooling. |
| **Use clear analogies** | Map unfamiliar onchain concepts to familiar ones. Walk sample user flows. |
| **Celebrate builders** | Builders aren't just developers — they're creators, artists, writers. Frequently highlight projects, people, and communities on Base. |
| **Act as a bridge** | Assume some readers are new to crypto. Use analogy and example to connect familiar tools to what's being built on Base. |

### 4.4 What to avoid when seeking Base amplification

| Risk | Explanation |
|---|---|
| **Financial language** | Avoid gains, profits, investments, high yields, leverage, lucrative opportunities. No investment advice, CTAs, or strategies. |
| **Specific token focus** | Focus on utility, gameplay, user experience — do not highlight specific tokens or token rewards (in copy or visuals). |
| **No project context** | Readers may encounter Humanbased for the first time. Don't post only about product features or milestones — establish context first. |
| **Unverified superlatives** | Avoid "the best", "the first", "maximize", "ensure" and similar unsubstantiated claims. |
| **Unauthorized partnership language** | Don't use "partnership" / "collaboration with Base" — use "launched on / built on / integrated with Base". |
| **Aggressive marketing** | No all-caps, no emoji storms, no discount-sale copy tactics. |
| **False or misleading information** | All information must be accurate, truthful, and non-misleading. |
| **Unsourced data** | Any statistic needs a source. |
| **Over-emphasizing the Coinbase connection** | Base is incubated by Coinbase but committed to decentralization. Focus on Base itself. |
| **Disparaging competitors** | Do not put down other projects. |
| **NSFW content** | Avoid offensive or explicit content across all channels, including the website. |
| **Unreleased product teasers** | Only tease what's immediately available. |
| **Incomplete posts** | Submissions for amplification must include all media (video/images/links) in the post itself. |
| **Unauthorized IP** | All IP in posts and on the site must be properly licensed. |
| **Non-compliant terms** | Any value-transfer application requires ToS that includes governing law and sanctions compliance. |
| **Geo restrictions (where applicable)** | US-restricted apps must clearly state "available to non-US users only" at the top of the post. |
| **KYC requirements (where applicable)** | KYC-gated apps must clearly state "available to KYC-verified / qualified users only" at the top of the post. |
